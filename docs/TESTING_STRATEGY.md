# Estrategia de Pruebas (Testing Strategy)

Este documento registra *por qué* la suite de tests está estructurada como está, no repite lo que el código ya dice. Referencia: [ADR-001-Stack-Tecnologico.md](ADR-001-Stack-Tecnologico.md).

## 1. Cobertura Exigida

Regla para tests nuevos: si un test de ingesta o de API no fuerza al menos un caso de "esto vino roto", la cobertura de esa ruta está incompleta (ver sección 3).

- **Backend**: toda ruta de ingesta de Excel (magic bytes, tamaño, Zip Bomb, XML, filas inválidas) y todo código de error HTTP expuesto por la API deben tener un test que fuerce ese caso límite explícitamente, no solo el camino feliz.
- **Frontend**: todo estado de error que la UI traduce a un banner (`ApiError.friendlyMessage`: 0/413/422/500) debe tener un test que mockee ese código y verifique el mensaje mostrado.
- Un reporte de bug no reproducible no se cierra solo con una lectura del código: se agrega un test de regresión que quede como evidencia verificable (ver "Foco del input de búsqueda" en la sección 4).

## 2. Estrategias por Capa

**Backend:**
- Lectura de los Excel de origen en streaming por lotes (`openpyxl`, modo `read_only`), sin cargar el archivo completo en memoria.
- Pruebas de integración sin base de datos: el sistema no tiene motor de datos (lee 4 Excel de `DATA_DIR` directamente, sin ORM), así que no se simula una capa de persistencia que no existe.
- **Límites de payload**: `MaxBodySizeMiddleware` rechaza con 413 antes de leer el body si `Content-Length` supera 1 MB (la API es de solo lectura, no recibe archivos por HTTP). `RateLimitMiddleware` limita a 60 requests/60s por IP (429 + `Retry-After`), ventana deslizante en memoria del proceso (si el backend corre con más de un worker Gunicorn, cada worker cuenta por separado — ceiling documentado en el código, no hay estado compartido tipo Redis). El Excel de origen tiene un tope propio: `MAX_EXCEL_SIZE_MB = 20` en `ingestion.py`, chequeado por tamaño en disco antes de que `openpyxl` toque el contenido.
- **Ingesta partial-success**: `validate_rows` (`ingestion.py`) descarta y reporta una fila inválida en vez de abortar el lote completo (`(DataFrame, list[str])`); solo un error de *archivo* completo (extensión, magic bytes, Zip Bomb, zip corrupto) sigue devolviendo 422. `_build_estados` (`api/muestras.py`) acumula los mensajes de checklist + area2 en `DashboardResponse.errores_validacion`, consumido por el frontend como banner no bloqueante (`Dashboard.tsx`).
- **Umbrales de fuzzy matching parametrizados**: `fuzzy_correct_threshold`/`fuzzy_search_threshold` viven en `core/config.py` (override por env var `FUZZY_CORRECT_THRESHOLD`/`FUZZY_SEARCH_THRESHOLD`) e inyectados en `correct_ids`/`search_by_code` desde `api/muestras.py`; los tests de `fuzzy_match.py` siguen pasando el `threshold` directo por parámetro (no dependen de `Settings`), y los tests de API pueden pisar `Settings` para verificar el comportamiento con umbrales distintos sin tocar el servicio.
- **Workers de Gunicorn acotados por memoria (`WEB_CONCURRENCY=1` por defecto)**: la nota de la línea anterior sobre "cada worker cuenta por separado" para el rate limiter aplica igual acá — el número de workers no es gratis en memoria. `backend/Dockerfile` arranca Gunicorn con `-w ${WEB_CONCURRENCY:-1}`, deliberadamente conservador: cada worker Uvicorn puede tener, durante una request, su propio `DataFrame` de Pandas con los 4 Excel cruzados en memoria a la vez. Esto trabaja en conjunto con la lectura en `read_only` de `openpyxl` (streaming por lotes, ver el primer punto de esta sección): el streaming acota cuánto pesa *un* DataFrame en construcción, y `WEB_CONCURRENCY=1` acota cuántos DataFrames de ese tamaño pueden coexistir en el proceso al mismo tiempo — juntos son lo que hace matemáticamente sostenible el límite de `BACKEND_MEM_LIMIT=512M` de `docker-compose.yml`. `--max-requests 1000` en el mismo `CMD` recicla el worker periódicamente para no acumular fragmentación/fugas de memoria en un proceso de vida larga. No hay un test de integración que mida RSS del contenedor (eso es infraestructura, no lógica de aplicación); la garantía es de diseño (1 worker, streaming, reciclado periódico), documentada acá y en el ADR-001.

**Frontend:**
- Pruebas de renderizado y debounce con React Testing Library (`DashboardPage.test.tsx`), incluyendo el test de regresión de foco/debounce (sección 4).
- **Invalidación de caché (React Query)**: el dashboard es de solo lectura, no hay mutaciones ni "invalidar tras escribir". `queryKey: ["muestras", debouncedQuery]` hace que cada búsqueda sea una entrada de caché distinta, sin invalidación manual. Reintentos selectivos (`queryClient.ts::shouldRetry`): un 4xx (422/413) no se reintenta porque reintentar no cambia una fila inválida en el Excel de origen; un fallo de red (status 0) o 5xx transitorio sí vale un reintento único. La frescura de datos se comunica vía `alertas_desfase` (`check_file_freshness`), no vía invalidación de query, porque los archivos fuente cambian fuera de la app y no hay evento del lado del cliente que dispare la actualización. Si en el futuro se agrega un endpoint de subida de archivo maestro, el punto de extensión correcto es un `onSuccess` que llame `queryClient.invalidateQueries({ queryKey: ["muestras"] })`.
- **`ErrorBoundary` vs. errores de red**: son dos mecanismos distintos, probados por separado. `DashboardPage` maneja el error que devuelve `useQuery` con un banner específico por código HTTP (cubierto en `DashboardPage.test.tsx`). `ErrorBoundary` (`frontend/src/components/ErrorBoundary.tsx`) cubre la clase de fallo que React Query no atrapa: una excepción lanzada durante el *render* (prop con forma inesperada, bug en un formateador) — sin él, ese error tira toda la app a blanco. Se probó forzando un `throw` en el render (`ErrorBoundary.test.tsx`) en vez de reusar los tests de error de red, porque las responsabilidades no se solapan.
- **AbortController**: `fetchDashboard`/`exportDashboard` aceptan un `AbortSignal` y lo pasan a `fetch`. `DashboardPage` no crea su propio `AbortController`: usa el `signal` que la propia `queryFn` de TanStack Query inyecta (`({ signal }) => fetchDashboard(debouncedQuery, signal)`), que React Query aborta automáticamente cuando `debouncedQuery` cambia o el componente se desmonta. Un `AbortError` se re-lanza tal cual en `api.ts` (no se envuelve en `ApiError`) para que React Query lo trate como cancelación, no como fallo a mostrar en un banner.

## 3. Inyección de Fallos (Fault Injection)

El sistema cruza cuatro archivos Excel que llegan de fuentes externas (áreas del laboratorio) fuera de nuestro control. La suite asume que esos archivos van a llegar rotos, y prueba explícitamente:

- **Extensión falseada**: un `.csv` renombrado a `.xlsx` (`test_integration.py::test_malicious_extension_is_rejected_before_parsing`). La defensa es el chequeo de *magic bytes* en `ingestion.assert_safe_excel_file`, no la extensión declarada.
- **Archivo sobredimensionado**: rechazo por tamaño antes de intentar abrir el workbook (`test_ingestion.py`), para no darle a un Zip Bomb la oportunidad de descomprimirse en memoria.
- **Zip Bomb (por tamaño descomprimido y por ratio de compresión)**: `test_assert_safe_excel_file_rejects_zip_bomb_by_uncompressed_size` y `..._by_compression_ratio` construyen un `.xlsx` cuyo tamaño en disco es mínimo pero cuyo `sharedStrings.xml` se infla a >200 MB al descomprimir. El chequeo de tamaño de archivo por sí solo no detecta esto (es exactamente el vector real de un Zip Bomb: comprime muy bien, pesa poco). La defensa lee `zipfile.ZipFile(...).infolist()` — metadata del directorio central, sin descomprimir nada — antes de dejar que openpyxl abra el archivo.
- **XXE (XML External Entity)**: no se prueba con un test dedicado porque la mitigación es a nivel de dependencia, no de lógica propia: `defusedxml` instalado hace que openpyxl (que chequea `DEFUSEDXML` en su propio `__init__`) reemplace el parser XML stdlib por uno que no resuelve entidades externas ni expande bombas de entidades, en todo el proceso. Verificar la mitigación es verificar que la dependencia esté instalada (`requirements.txt`), no escribir un test de comportamiento.
- **Input adversarial en la búsqueda difusa**: `search_by_code` sanitiza el `q` de la request (caracteres de control removidos, tope de 200 caracteres) antes de pasarlo a `thefuzz`/`rapidfuzz`, cubierto en `test_fuzzy_search.py` (query con caracteres de control, query de 10.000 caracteres, query compuesta solo por caracteres de control) y en `test_hardening.py` (`test_malformed_query_param_...`, `test_excessively_long_query_param_...`) a nivel de endpoint completo.
- **Zip corrupto con magic bytes válidos**: un archivo que arranca con `PK\x03\x04` (pasa el chequeo de magic bytes) pero cuyo contenido está truncado (`test_assert_safe_excel_file_rejects_corrupt_zip_with_valid_magic_bytes`) — la extensión y los primeros bytes no garantizan que el zip sea válido.
- **Filas inválidas (partial success)**: una fila que no valida contra el schema Pydantic (`ChecklistRow`/`AnalisisRow`) se descarta y se reporta en `errores_validacion`, sin abortar el resto del lote (`test_ingestion.py::test_validate_rows_skips_invalid_row_and_keeps_the_rest`, `test_api.py::test_get_muestras_reports_partial_success_when_a_checklist_row_is_invalid`). El caso límite de un lote 100% inválido (`test_validate_rows_returns_empty_result_when_every_row_is_invalid`) confirma que no crashea, solo devuelve un resultado vacío. Un error de *archivo* completo (extensión falsa, magic bytes, Zip Bomb) sigue siendo abortivo con 422 — no hay "resto del lote" que rescatar cuando el archivo entero es ilegible.
- **Excepciones no controladas en el handler**: se fuerza un `RuntimeError` con un mensaje que nunca debería llegar al cliente, y se verifica que el 500 devuelto no contenga ni el mensaje ni el traceback (`test_hardening.py::test_unhandled_exception_does_not_leak_internals`).
- **Payloads malformados/excesivos contra `api/muestras.py`**: además del límite de `Content-Length` (413), `test_hardening.py` envía un query param `q` con caracteres de inyección/HTML y un query param de 50.000 caracteres a `/api/muestras/buscar`, verificando que ninguno de los dos crashea el endpoint (200, no 500) — `search_by_code` opera sobre `str` de Python sin interpolarlo en ningún comando/query, así que no hay superficie de inyección real, pero el test deja documentado que un input adversarial no rompe el servicio.
- **Fallas de red en el cliente**: `DashboardPage.test.tsx` mockea `fetchDashboard`/`exportDashboard` rechazando con cada código relevante (0 = sin conexión, 413, 422, 500) y verifica que la UI muestre un banner legible en vez de pantalla en blanco o el error crudo.

## 4. Decisiones Históricas y Deuda Técnica

- **Simular caídas de base de datos**: descartado. El proyecto no tiene base de datos (lee 4 Excel de `DATA_DIR`, sin ORM ni motor de datos en `config.py`). No se agregó una capa de base de datos solo para poder simular su caída.
- **Zod para validación de esquemas**: descartado. Se mantuvo el type guard hecho a mano (`isDashboardResponse`/`isMuestraEstado`), reforzado campo por campo (cada `MuestraEstado` validado: id string, estado dentro del enum válido, arrays de strings) — agregar Zod habría sido una dependencia estructural nueva para algo que un guard sin dependencias ya cubre por completo.
- **`throwOnError: true` global en `queryClient.ts`**: descartado. `DashboardPage` ya maneja el error localmente con un banner específico por código HTTP; ponerlo global habría reemplazado ese mensaje por el fallback genérico del `ErrorBoundary`.
- **Interceptores tipo Axios**: descartado. El stack fijado en el ADR-001 no usa Axios; `api.ts` ya centraliza la traducción de errores en un wrapper (`fetchJson`/`ApiError`), equivalente correcto para `fetch`.
- **Parseo de fechas ISO en `api.ts`/`muestra.ts`**: descartado. El contrato de datos (`schemas.py`/`muestra.ts`) no tiene ningún campo de fecha — todo es `str`/`list[str]`/enum de estado.
- **`useMuestrasMutation`, invalidación de caché "tras subida" y Optimistic UI**: descartado. Esta API es de solo lectura (`GET /api/muestras`, `/buscar`, `/exportar`); no existe endpoint de subida ni mutación en el sistema.
- **Partial success en `validate_rows` vs. abortar el lote (2026-07-18)**: se cambió el comportamiento deliberado de 1.1.0 ("una fila rota aborta todo el archivo con 422") a partial-success ("una fila rota se descarta y se reporta, el resto se procesa"), por pedido explícito de una nueva ronda de auditoría. El test `test_get_muestras_rejects_checklist_with_missing_required_column` se reescribió como `test_get_muestras_reports_partial_success_when_a_checklist_row_is_invalid` (200 + `errores_validacion`, no 422) porque el comportamiento que verificaba ya no es el esperado. `test_known_http_errors_still_get_their_real_status_code` (`test_hardening.py`) se migró de "fila inválida" a "archivo con magic bytes inválidos" como disparador del 422, para no perder cobertura de que un `HTTPException` explícito no queda atrapado por el manejador genérico.
- **Auditoría de infraestructura/red (Dockerfiles, `nginx.conf`, `middleware.py`, `AbortController`, interceptor de errores en `api.ts`) — sin cambios (2026-07-18)**: una ronda de auditoría pidió explícitamente estos ocho puntos; los ocho ya estaban implementados desde 1.1.0–1.3.0 (usuario `appuser` + Gunicorn/UvicornWorker en `backend/Dockerfile`, multi-stage + `nginx-unprivileged` en `frontend/Dockerfile`, CSP/`X-Content-Type-Options`/`X-Frame-Options` en `nginx.conf`, `sanitized_error_handler` en `main.py`, `signal` de React Query pasado a `fetch` en `api.ts`, `ApiError`/`fetchJson` como interceptor centralizado). No se tocó código; se dejó registrado en el CHANGELOG para que la auditoría no se repita sobre el mismo punto sin contexto.
- **`backend/Dockerfile` en `python:3.12-slim`, no Alpine/distroless (2026-07-18)**: una ronda de auditoría pidió explícitamente Alpine o distroless. Se mantuvo `slim` porque `pandas`/`numpy` no publican wheels para musl (Alpine), lo que forzaría compilar ambos desde código fuente en el build — el mismo tipo de problema de disponibilidad de wheels que ya complicó fijar versiones para Python 3.14 (ver `ADR-001`, sección "Consecuencias"). `slim` ya es multi-stage y corre como `appuser` sin privilegios; no hay superficie de ataque adicional relevante que Alpine resuelva para este caso.
- **`queryClient.invalidateQueries()` tras mutaciones de lotes (2026-07-18)**: descartado, no aplica. La API es de solo lectura (`GET /api/muestras`, `/buscar`, `/exportar`); no existe endpoint `POST`/`PUT`/`DELETE` que mute un lote de muestras en `api/muestras.py`. Ya se había descartado por la misma razón en 1.2.0/1.4.0 para casos equivalentes (`useMuestrasMutation`, Optimistic UI).
- **Deuda técnica identificada (2026-07-18)**: ausencia de pruebas E2E de la SPA (tipo Cypress/Playwright) y de pruebas formales de carga (load testing) sobre el chunking de `ingestion.py` con archivos cercanos al límite de `MAX_EXCEL_SIZE_MB`/`MAX_UNCOMPRESSED_MB`. No se abordan en esta iteración por alcance; quedan para una versión posterior.
- **Foco del input se perdía en cada letra de una búsqueda con latencia real (regresión, 2026-07-18)**: reportado en pruebas manuales del usuario. Causa: cada `debouncedQuery` nuevo era un `queryKey` de React Query sin datos cacheados (`isPending: true`), y `DashboardPage.tsx` reemplazaba toda la vista (input incluido) por `<p>Cargando...</p>` mientras la respuesta viajaba. El test de regresión de foco existente (sección histórica de "foco del input de búsqueda", más abajo) no lo detectaba porque su mock resolvía `fetchDashboard` en el mismo tick — sin ventana de tiempo real donde `isPending` fuera observable como `true`. Fix: `placeholderData: keepPreviousData` en el `useQuery`. Test nuevo (`DashboardPage.test.tsx`, "keeps the search input mounted and focusable while a debounced query is still in flight") que deja la segunda consulta deliberadamente sin resolver para poder observar el estado intermedio, verificado manualmente que falla sin el fix.
- **Workers de Gunicorn conservadores sobre alto paralelismo (2026-07-18)**: `WEB_CONCURRENCY=1` por defecto no es un descuido de performance, es la decisión correcta para esta carga de trabajo. Cada request de `/api/muestras` hace matching difuso (`thefuzz`/`rapidfuzz`, CPU-bound) y manipula varios `DataFrame` de Pandas simultáneamente (checklist + área 2 + el cruce), cada uno con su propio pico de memoria durante la construcción del resultado. Escalar con más workers multiplicaría ese pico por worker, no lo paraleliza de forma barata (a diferencia de un endpoint I/O-bound típico, donde más workers cuestan poca memoria adicional). Con el `BACKEND_MEM_LIMIT=512M` ya fijado, subir `WEB_CONCURRENCY` sin subir el límite de memoria en la misma proporción es la forma más directa de provocar un OOM kill del contenedor.
- **`maxLength` en el frontend vs. truncamiento estricto en el backend (defensa en profundidad, 2026-07-18)**: `Dashboard.tsx` limita el `<input>` de búsqueda a 200 caracteres (`maxLength={MAX_QUERY_LENGTH}`), pero esa es una restricción de UX, no de seguridad — cualquier cliente que no sea el navegador (curl, un script) puede mandar un `q` de cualquier longitud directo a la API. La regla real vive exclusivamente en el backend (`_sanitize_query`, `fuzzy_match.py`), que trunca sin importar qué mandó el cliente. Se decidió no depender del `maxLength` del HTML como control de seguridad ni testear el frontend como si lo fuera; el test de `Dashboard.test.tsx` solo verifica que el atributo esté presente (mejor UX), y el test real de la garantía de seguridad (`test_search_by_code_caps_an_oversized_query_...`) vive en `test_fuzzy_search.py`, contra el backend.
- **Log (`warning`) en vez de excepción al truncar una query oversized (2026-07-18)**: `_sanitize_query` registra el evento con `logger.warning` y sigue procesando la query truncada, en vez de lanzar una excepción que le devolviera un error al usuario. Razón: una query de más de 200 caracteres no es un ataque confirmado ni un estado inválido del sistema — es, la mayoría de las veces, un paste accidental o un usuario probando el límite — y el sistema puede seguir operando correctamente con la versión truncada (sigue siendo una búsqueda válida). Cortar la experiencia con un 4xx ahí penalizaría al usuario legítimo por un caso que el sistema ya sabe manejar con seguridad; el `warning` le da al equipo de seguridad la visibilidad de auditoría sin sacrificar disponibilidad. Contrasta con los casos que sí devuelven un error explícito (413 de `MaxBodySizeMiddleware`, 422 de un Excel de origen ilegible): ahí no hay una interpretación segura del input, así que abortar es la única opción correcta.
- **Regresión: foco del input de búsqueda y debounce efectivo** — Se reportó como bug crítico que el input de búsqueda perdía el foco por tecla y disparaba un fetch por cada carácter. Al revisar el código (`Dashboard.tsx` + `DashboardPage.tsx` + `useDebounce.ts`) no se encontró el defecto descripto: el input vive en un componente de función estable, sin recrearse ni cambiar de `key` entre renders, y el `queryKey` de React Query solo cambia cuando `debouncedQuery` cambia (300ms después de la última tecla). En vez de cerrar el reporte solo con esa lectura del código, se agregó un test de regresión (`DashboardPage.test.tsx`, "keeps focus on the search input and debounces the fetch while typing") que escribe carácter por carácter con timers falsos, verifica que `document.activeElement` sigue siendo el input después de cada tecla, y que `fetchDashboard` se llama una sola vez recién al vencer el debounce. Si en el futuro alguien reintroduce el bug (ej. moviendo el input a un componente que se recrea por render, o pasando `debouncedQuery` en vez de `query` al `value` del input), este test debería fallar y señalarlo.
